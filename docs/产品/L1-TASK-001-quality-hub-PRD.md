# L1-TASK-001 Quality-Hub 项目质量管理平台 产品需求文档

## 1. 需求背景

### 来源
团队在多项目并行开发中，缺乏统一的质量管理工具来追踪功能点覆盖情况、管理测试用例、执行 E2E 自动化测试并生成质量报告。

### 现状
- 功能点散落在 PRD 和 Issue 中，无法统一追踪覆盖率
- 测试用例缺乏结构化管理，与功能点的关联不清晰
- E2E 测试执行依赖手动操作，无法在部署后自动触发
- 质量数据无可视化看板，团队对项目质量状态缺乏全局视角

### 痛点
1. 无法快速回答"哪些功能点还没有测试用例覆盖"
2. 部署后需要手动触发测试，容易遗漏
3. AI 角色无法通过 API 自动创建/更新测试用例
4. 缺乏历史趋势数据，无法评估质量改善效果

---

## 2. 目标用户与场景

### 目标用户
| 用户角色 | 使用场景 |
|---------|---------|
| 产品经理 | 查看功能点覆盖率，确认需求是否被充分测试 |
| 测试人员 | 管理测试用例，触发执行，查看报告 |
| 开发者 | 部署后查看自动化测试结果 |
| AI 角色 | 通过 API 自动创建功能点、测试用例、触发执行 |
| 项目负责人 | 通过看板了解整体质量状态和趋势 |

### 核心场景
1. **新需求入库**：PM 创建项目 → 从 PRD/Issue 拆出功能点 → AI/测试人员编写用例
2. **日常测试**：选择功能点/用例 → 触发 Midscene 执行 → 查看结果
3. **部署验证**：部署完成 → API 触发全量/增量执行 → 结果回写 → 看板更新
4. **质量评估**：查看覆盖率/通过率/趋势图 → 识别薄弱环节

## 3. 功能描述

### 3.1 项目管理

**核心流程：**
- 创建项目（名称、描述、关联 GitHub 仓库 URL）
- 项目列表展示（名称、仓库、功能点数、覆盖率概览）
- 项目详情页入口

**边界条件：**
- 项目名称唯一
- GitHub 仓库 URL 格式校验
- 删除项目需确认（级联删除功能点、用例、执行记录）

### 3.2 功能点管理

**核心流程：**
- 在项目下创建功能点（名称、描述、来源 Issue 编号）
- 功能点状态流转：待覆盖 → 部分覆盖 → 已覆盖
- 功能点列表支持按状态筛选

**边界条件：**
- 功能点名称在项目内唯一
- 状态根据关联用例数量自动计算（0 条=待覆盖，有用例但未全部通过=部分覆盖，全部通过=已覆盖）
- 来源 Issue 编号为可选字段

### 3.3 测试用例管理（核心模块）

**核心流程：**
- 在功能点下创建测试用例
- 用例字段：标题、描述、前置条件、测试步骤（有序列表）、预期结果、关联 Midscene 脚本路径
- 支持增删查改（前端界面 + RESTful API）
- AI 角色通过 API 批量创建/更新用例

**边界条件：**
- 用例标题在功能点内唯一
- Midscene 脚本路径为可选（未关联时只能手动标记结果）
- 删除用例需确认
- API 支持批量创建（单次最多 50 条）

### 3.4 执行管理

**核心流程：**
- 创建执行任务（选择范围：整个项目 / 指定功能点 / 单条用例）
- 调用 Midscene.js 执行引擎运行关联脚本
- 执行结果回写：通过 / 失败 / 跳过（无脚本）
- 失败时保存截图和错误信息

**触发方式：**
- 手动触发：前端界面点击执行
- API 触发：POST /api/runs（部署后自动调用）

**边界条件：**
- 同一项目同时只允许一个执行任务运行
- 执行超时：单条用例 60 秒，整体任务 30 分钟
- 无 Midscene 脚本的用例标记为"跳过"

### 3.5 报告看板

**核心指标：**
- 覆盖率：有用例的功能点数 / 总功能点数
- 通过率：最近一次执行中通过的用例数 / 总执行用例数
- 历史趋势：按天/周展示覆盖率和通过率变化曲线

**展示维度：**
- 全局看板（所有项目汇总）
- 项目级看板（单项目详情）

**边界条件：**
- 无执行记录时显示"暂无数据"
- 趋势图最多展示最近 90 天

## 4. 验收标准

### 4.1 项目管理
- [ ] 可创建项目并填写名称、描述、GitHub 仓库 URL
- [ ] 项目列表正确展示所有项目及概览数据
- [ ] 项目名称重复时给出错误提示
- [ ] 删除项目时弹出确认框，确认后级联删除

### 4.2 功能点管理
- [ ] 可在项目下创建功能点（名称、描述、来源 Issue）
- [ ] 功能点状态根据用例覆盖情况自动计算
- [ ] 支持按状态筛选功能点列表
- [ ] API 创建功能点返回正确响应

### 4.3 测试用例管理
- [ ] 可在功能点下创建用例（含所有字段）
- [ ] 前端界面支持用例的增删查改
- [ ] API 支持单条和批量创建用例（最多 50 条）
- [ ] API 支持更新用例
- [ ] Midscene 脚本路径为可选字段

### 4.4 执行管理
- [ ] 可手动触发执行（项目级/功能点级/单条用例）
- [ ] API 触发执行正常工作
- [ ] 执行结果正确回写（通过/失败/跳过）
- [ ] 失败时保存截图和错误信息
- [ ] 同一项目并发执行时拒绝第二个任务

### 4.5 报告看板
- [ ] 覆盖率计算正确
- [ ] 通过率计算正确
- [ ] 历史趋势图正确展示（最近 90 天）
- [ ] 无数据时显示友好提示

## 7. 非功能性要求

### 性能
- API 响应时间 < 500ms（常规 CRUD）
- 看板页面加载 < 2s
- 执行任务启动延迟 < 5s

### 安全
- API 输入参数校验（防注入）
- 文件路径校验（Midscene 脚本路径不允许目录穿越）

### 兼容性
- 前端支持 Chrome 最新版
- API 兼容 RESTful 规范，方便 AI 角色调用

### 可维护性
- 代码注释使用中文
- API 自动生成文档（FastAPI /docs）

## 8. 风险与备选方案

| 风险 | 影响 | 备选方案 |
|------|------|---------|
| Midscene.js 执行环境依赖复杂 | 执行模块延期 | 先实现手动标记结果，Midscene 集成作为 M2 交付 |
| SQLite 并发写入限制 | 多用户同时操作可能冲突 | 初期单用户使用足够；后续可迁移 PostgreSQL |
| AI 批量创建用例数据质量不可控 | 垃圾数据 | API 层做字段校验 + 长度限制 |
