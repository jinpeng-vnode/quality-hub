# Quality Hub 体验优化建议 — 新用户视角

> 测试日期：2026-05-27
> 测试角色：首次使用平台的 QA 工程师
> 测试地址：https://quality-hub.todonot.com/
> 操作路径：首页 → 创建项目 → 功能点管理 → 测试用例 → 执行管理 → 报告看板

---

## 一、问题总览

| 优先级 | 数量 | 说明 |
|--------|------|------|
| P0（阻塞） | 3 | 功能缺失或逻辑错误，用户无法完成核心流程 |
| P1（严重） | 5 | 信息展示错误，严重影响使用体验 |
| P2（中等） | 7 | 体验不佳，增加用户认知负担 |
| P3（轻微） | 4 | 细节打磨，提升专业感 |

---

## 二、P0 阻塞问题

### 2.1 手动执行模式无法标记用例结果

**现象**：触发"手动模式"执行后，用例状态显示"等待中"，但没有任何操作按钮让用户标记通过/失败。用户无法完成手动测试流程。

**影响**：手动测试流程完全断裂，核心功能不可用。

**建议**：在执行详情的用例列表中，为"等待中"状态的用例提供"标记通过"/"标记失败"/"跳过"操作按钮。

### 2.2 脚本模式执行结果逻辑矛盾

**现象**：脚本模式执行后，整体状态显示"通过"，但通过率为"0.0%"，用例状态为"跳过"。三个信息互相矛盾。

**影响**：用户无法判断执行是否成功，数据不可信。

**建议**：
- 如果所有用例被跳过，整体状态应为"跳过"或"未执行"，而非"通过"
- 明确说明用例被跳过的原因（如"该用例类型为手动，脚本模式不支持"）
- 通过率计算应排除跳过的用例，或单独展示

### 2.3 触发执行无确认步骤

**现象**：点击"手动模式"或"脚本模式"后直接创建执行记录，没有确认对话框，也没有让用户选择要执行哪些用例。

**影响**：用户可能误触发执行，且无法选择性执行部分用例。

**建议**：
- 点击后弹出确认对话框，显示将要执行的用例列表
- 允许用户勾选要执行的用例
- 提供"全选/取消全选"快捷操作

## 三、P1 严重问题

### 3.1 功能点下拉显示 UUID 而非名称

**现象**：创建测试用例时，"所属功能点"下拉选项显示完整 UUID（如 `4f240c8f-a7b1-4a28-b436-72b324bf0b3c`），选中后同时显示 UUID 和名称。

**影响**：用户无法通过 UUID 识别功能点，严重影响操作效率。

**建议**：下拉选项只显示功能点名称，value 使用 UUID 但对用户不可见。

### 3.2 执行详情中用例 ID 显示完整 UUID

**现象**：执行详情的"用例结果"表格中，用例 ID 列显示完整 UUID（如 `bbd5e2b8-1698-4240-8204-3ad2fd56748c`）。

**影响**：用户无法通过 UUID 识别是哪个用例，需要手动对照。

**建议**：显示用例标题，或至少显示"标题 (短ID)"格式。

### 3.3 功能点来源显示英文

**现象**：功能点列表中"来源"列显示英文 `manual`，而非中文"手动"。

**影响**：中英文混杂，界面不一致。

**建议**：所有枚举值统一显示中文标签（manual→手动, prd→PRD, issue→Issue）。

### 3.4 时间格式不友好

**现象**：执行详情中的开始/结束时间显示完整 ISO 格式（如 `2026-05-27T00:37:25.551407+00:00`）。

**影响**：信息密度过高，难以快速阅读。

**建议**：使用友好格式如 `2026-05-27 00:37:25`，或相对时间如"刚刚"/"5分钟前"。

### 3.5 按钮文案中英文不一致

**现象**：
- 新建项目/功能点对话框：Cancel / OK（英文）
- 新建测试用例抽屉：取消 / 保存（中文）

**影响**：界面风格不统一，降低专业感。

**建议**：全部统一为中文"取消"/"确定"或"取消"/"保存"。

## 四、P2 中等问题

### 4.1 无面包屑导航

**现象**：进入项目后，页面没有面包屑显示当前位置（如"项目管理 > 体验测试项目 > 功能点管理"）。

**影响**：用户不知道当前在哪个项目中操作，多项目场景下容易混淆。

**建议**：在页面标题上方添加面包屑导航。

### 4.2 空状态无引导

**现象**：所有页面的空状态只显示"暂无XXX"文字和图标，没有操作引导。

**影响**：新用户不知道下一步该做什么。

**建议**：空状态增加引导文案和操作按钮，例如：
- "暂无项目，点击「新建项目」开始管理你的测试质量"
- "暂无功能点，从 PRD 或 Issue 导入功能点，开始追踪测试覆盖率"

### 4.3 功能点状态"待覆盖"含义不明

**现象**：功能点创建后状态为"待覆盖"，但没有说明"覆盖"的定义。

**影响**：用户不理解何时状态会变化，不知道需要做什么才能"覆盖"。

**建议**：
- 添加状态说明 tooltip（如"关联测试用例且执行通过后变为已覆盖"）
- 或在功能点详情中展示覆盖条件

### 4.4 测试用例列表缺少关键列

**现象**：测试用例表格没有"所属功能点"列。

**影响**：当用例数量多时，无法快速判断用例归属。

**建议**：增加"所属功能点"列，或支持按功能点分组显示。

### 4.5 功能点列表无快捷操作

**现象**：功能点列表操作列只有"编辑"和"删除"，没有"创建用例"或"查看关联用例"入口。

**影响**：用户需要切换到测试用例页面才能为功能点创建用例，操作路径长。

**建议**：在操作列增加"创建用例"快捷按钮，或点击功能点行展开关联用例列表。

### 4.6 ID 列显示 UUID 前8位

**现象**：功能点和测试用例的 ID 列显示 UUID 前8位（如 `4f240c8f`）。

**影响**：对用户无意义，占用表格空间。

**建议**：
- 使用自增序号（如 FP-001, TC-001）
- 或隐藏 ID 列，改为点击行展开详情时显示

### 4.7 报告看板覆盖率定义不明

**现象**：报告看板显示"覆盖率 0.0%"，但已创建了功能点和关联用例。

**影响**：用户不理解覆盖率的计算逻辑。

**建议**：在覆盖率数字旁添加说明（如"已覆盖功能点数 / 总功能点数"），并明确"覆盖"的判定条件。

## 五、P3 轻微问题

### 5.1 页面标题为 "frontend"

**现象**：浏览器标签页标题显示 "frontend" 而非产品名称。

**建议**：改为 "Quality Hub" 或 "QHub - 项目质量管理平台"。

### 5.2 创建成功无 Toast 提示

**现象**：创建项目、功能点成功后没有明显的成功反馈。

**建议**：添加 Toast 提示"创建成功"。

### 5.3 左侧导航无项目名称标识

**现象**：进入项目后，左侧导航只显示功能菜单，没有当前项目名称。

**建议**：在导航顶部（QHub logo 下方）显示当前项目名称，并提供返回项目列表的入口。

### 5.4 缺少新手引导

**现象**：首次使用没有任何引导流程，用户需要自行摸索。

**建议**：首次登录时提供简单的引导流程（3-5步），介绍核心概念和操作路径：创建项目 → 添加功能点 → 编写用例 → 触发执行 → 查看报告。

---

## 六、整体建议

### 6.1 核心流程优化

当前用户操作路径：
```
创建项目 → 创建功能点 → 创建用例(关联功能点) → 触发执行 → 查看报告
```

建议优化为更连贯的流程：
- 创建功能点后，提供"立即创建用例"的快捷入口
- 创建用例后，提供"立即执行"的快捷入口
- 执行完成后，自动跳转到报告页面

### 6.2 信息架构优化

- 所有 UUID 对用户不可见，使用有意义的标识符
- 所有枚举值统一使用中文显示
- 所有时间使用友好格式
- 所有按钮文案统一语言

### 6.3 概念说明

平台使用了一些专业概念（功能点、覆盖率、来源），建议：
- 在首次使用时提供概念说明
- 在相关字段旁添加 tooltip 解释
- 提供帮助文档入口
